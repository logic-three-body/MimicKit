# MimicKit 视觉复现验收清单（论文/README 用户视角）

## 1. 先说结论口径

1. 目标应定义为“复现同类运动效果”，不是“逐帧一致”。
2. README 的 GIF/图片通常是精选模型与镜头，零训练复现会受随机种子、后端、硬件影响。
3. 推荐先跑 `8h/case` 全量拿首轮结果，再对关键案例补到 `24h/case`。

## 2. 验收分级（建议统一用这套）

1. `L1 流程通过`：
`train -> test -> visualize` 全部可跑通，日志无致命错误，模型可加载。
2. `L2 视觉可用`：
角色能稳定完成目标动作，明显不像随机/崩溃策略。
3. `L3 README/论文级观感`：
动作风格、连贯性、稳定性接近 README 展示（允许细节差异）。

建议以 `L2` 作为“已复现可交付”，`L3` 作为“高质量复现”目标。

## 3. 对齐比较对象（避免误判）

每个案例都至少对比两组模型：

1. 你的训练模型：
`output/train/<run>/model.pt`
2. 仓库预训练模型：
`data/models/*.pt`

说明：预训练模型是 README 效果的直接参照，不要只看“单次可视化是否好看”。

## 4. 单案例标准流程（建议固定）

1. 训练完成后先做无渲染测试：
```bash
python mimickit/run.py \
  --arg_file args/<case>.txt \
  --mode test \
  --visualize false \
  --num_envs 1 \
  --model_file output/train/<run>/model.pt
```

2. 做可视化测试（同机位观察）：
```bash
python mimickit/run.py \
  --arg_file args/<case>.txt \
  --mode test \
  --visualize true \
  --num_envs 1 \
  --model_file output/train/<run>/model.pt
```

3. 对同案例预训练模型重复第 2 步。

4. 记录三项主观评分（1-5）：
`稳定性`、`动作风格一致性`、`任务完成度`。

5. 验收判定：
总分 `>=10/15` 记为 `L2`；
总分 `>=13/15` 且无明显抖动/摔倒，记为 `L3`。

## 5. 客观门槛（最小化）

1. `test` 阶段必须正常结束（返回码 0）。
2. 至少有有效评估日志输出（如 Mean Return / episode 指标行）。
3. 可视化阶段不能持续报图形致命错误（例如重复 GL/Warp fatal 后无画面）。

## 6. 预算策略（8h -> 24h）

1. 第一轮：全案例 `8h/case`
- 目标：快速拿到全量 `L1/L2` 分布和失败清单。
- 产出：每个案例至少有一次可对比的 test/viz 结果或明确失败原因。

2. 第二轮：定向补到 `24h/case`
- 适用：`final_ok=0`、或视觉评分未达预期（尤其目标是 `L3`）。
- 常见优先对象：`pi_plus`、多卡通信敏感案例、风格尚未收敛案例。

3. 这比直接全量 24h 更高效
- 先覆盖再加深，整体迭代更快。
- 把 24h 预算集中在真正需要追加训练的案例上。

## 7. 复现实务建议

1. 固定后端、版本、参数再比较。
2. 先追求流程稳定，再追求观感最优。
3. 对失败案例保留日志和模型快照，避免重复试错。
4. 使用同一 root 做 8h->24h 续跑，保持 checkpoint 连续性。

## 8. 结果记录模板

建议每个案例记录：

1. `case`
2. `engine`
3. `train_budget`
4. `model_path`
5. `L1/L2/L3`
6. `稳定性(1-5)`
7. `风格一致性(1-5)`
8. `任务完成度(1-5)`
9. `对比预训练模型结论`
10. `是否建议追加训练时长`

这样可以把“我看起来像不像论文效果”变成可追踪、可复盘的验收数据。
